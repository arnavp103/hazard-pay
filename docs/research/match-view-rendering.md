# Match-view rendering: game engine vs hand-rolled canvas vs DOM

- **Date:** 2026-07-20
- **Question:** For the browser pixel-art auto-battler match view inside a TanStack Start (SPA-mode) React 19 app: game engine (PixiJS, Phaser, Excalibur, Kaplay), hand-rolled Canvas 2D/WebGL, or no canvas at all (DOM/CSS transforms)? Deliver a recommendation for the first prototype (two pixel characters with idle animation) plus the criteria that would trigger revisiting the choice.
- **Context:** Server-authoritative game — the client renders state received over the network (tick results), so the render layer needs a scene graph, sprites, sprite-sheet animation, and tweening/interpolation; not physics or input-heavy game loops. Entity counts are auto-battler scale (dozens, not thousands), so DX and React interop outweigh raw perf. HUD/overlays/tooltips will be React components layered over the scene. Repo: TypeScript, pnpm monorepo, raw-TS packages, Vite via TanStack Start, ESLint-only formatting.
- **Method:** Primary sources only — official docs, GitHub repos (releases, issue activity), npm registry/downloads API, MDN/W3C/web.dev, react.dev, playwright.dev, vitest.dev, tanstack.com. Every claim carries its source URL. Versions checked 2026-07-20: pixi.js 8.19.0, @pixi/react 8.0.5, phaser 4.2.1, excalibur 0.32.0, kaplay 3001.0.19 (v4000 still alpha).

## 1. Cross-cutting: what "crisp pixel art in a browser" actually requires

These constraints apply to every candidate, so they are the baseline any option must clear.

- **CSS `image-rendering: pixelated`** is the DOM/canvas-element-level switch: "scaled with the 'nearest neighbor' or similar algorithm to the nearest integer multiple of the original image size, then uses smooth interpolation to bring the image to the final desired size" ([MDN image-rendering](https://developer.mozilla.org/en-US/docs/Web/CSS/image-rendering)). Defined in CSS Images Level 3 (Candidate Recommendation Draft; Level 4 does not touch it — [spec §5.2](https://drafts.csswg.org/css-images-3/#the-image-rendering)); Baseline "widely available" since January 2020 (MDN, same page).
- **Canvas 2D smoothing** is separate: `ctx.imageSmoothingEnabled` defaults to `true` and must be set `false` for pixel art ([MDN imageSmoothingEnabled](https://developer.mozilla.org/en-US/docs/Web/API/CanvasRenderingContext2D/imageSmoothingEnabled)). The canonical setup is a low-res backing canvas + CSS-scaled display size + `image-rendering: pixelated`, keeping every scale factor an integer ([MDN Games: Crisp pixel art look](https://developer.mozilla.org/en-US/docs/Games/Techniques/Crisp_pixel_art_look)).
- **devicePixelRatio is the universal spoiler.** Non-integer DPR (1.25, 2.625 from OS scaling) means CSS pixels don't align with device pixels, so "certain pixels may be drawn larger than others, resulting in a non-uniform appearance" — shimmer that MDN states is not fully solvable in CSS ([MDN image-rendering](https://developer.mozilla.org/en-US/docs/Web/CSS/image-rendering); [MDN Crisp pixel art look](https://developer.mozilla.org/en-US/docs/Games/Techniques/Crisp_pixel_art_look)). Sharp canvas rendering means sizing the backing store by DPR (`canvas.width = rect.width * dpr`) ([web.dev High DPI Canvas](https://web.dev/articles/canvas-hidipi)), ideally via `ResizeObserver`'s `devicePixelContentBoxSize` for exact 1:1 mapping ([web.dev devicePixelContentBox](https://web.dev/device-pixel-content-box/)), and re-checking on DPR change via the documented `matchMedia('(resolution: …dppx)')` re-subscription pattern ([MDN devicePixelRatio](https://developer.mozilla.org/en-US/docs/Web/API/Window/devicePixelRatio)).
- **TanStack Start SPA mode removes the SSR question.** In SPA mode "only the root route gets prerendered" into a shell; all other routes render client-side only ([TanStack Start SPA mode](https://tanstack.com/start/latest/docs/framework/react/guide/spa-mode.md)). The match route is therefore guaranteed client-only — a `useRef`/`useEffect` canvas mount needs no extra client-only guards.

## 2. Candidates

### 2.1 PixiJS v8 (+ @pixi/react)

**Health.** pixi.js 8.19.0 on npm; ~monthly minors (8.16.0 Feb 2026 → 8.19.0 Jun 2026); repo pushed 2026-07-19; 47.8k stars; TypeScript-authored, dual ESM/CJS with first-party `.d.ts` ([npm](https://registry.npmjs.org/pixi.js/latest); [pixijs.com/blog](https://pixijs.com/blog); [repo](https://github.com/pixijs/pixijs)). 867k weekly downloads — by far the most-used option surveyed ([npm downloads API](https://api.npmjs.org/downloads/point/last-week/pixi.js)). v8 renders via WebGL or WebGPU, and since 8.16.0 also has an experimental Canvas 2D renderer (`app.init({ preference: 'canvas' })`) covering "sprites, graphics, text, and basic filters" for "environments where a GPU context isn't available" ([8.16.0 release post](https://pixijs.com/blog/8.16.0)). Bundle: 251 KB gzip ([bundlephobia](https://bundlephobia.com/package/pixi.js)).

**Pixel art.** Global nearest-neighbor via `TextureStyle.defaultOptions.scaleMode = 'nearest'` (set before any texture is created), or per-texture `scaleMode` ([maintainer answer, discussion #11018](https://github.com/pixijs/pixijs/discussions/11018); [textures guide](https://pixijs.com/8.x/guides/components/textures)). Application options include `roundPixels`, `resolution` ("the resolution / device pixel ratio of the renderer"), `autoDensity` ("resizes renderer view in CSS pixels to allow for resolutions other than 1"), and `antialias` ([ApplicationOptions API](https://pixijs.download/release/docs/app.ApplicationOptions.html)) — i.e. DPR handling is built in rather than hand-rolled.

**Animation/tweening.** `AnimatedSprite` + `Spritesheet` (TexturePacker-shaped JSON; Aseprite exports compatible frame data) cover sprite-sheet playback ([AnimatedSprite API](https://pixijs.download/dev/docs/scene.AnimatedSprite.html); [sprite-sheet guide](https://pixijs.io/guides/basics/sprite-sheets.html)). There is **no built-in tween system** — the `Ticker` guide documents frame callbacks only ([ticker guide](https://pixijs.com/8.x/guides/components/ticker)); tweening is an external lib or ~30 lines of lerp on the ticker. For a server-tick interpolator we would write that lerp ourselves regardless, so this is a non-cost.

**React story.** Two viable patterns:
1. *Plain imperative mount:* create `Application` in `useEffect`, append its canvas to a ref'd div, destroy on cleanup — exactly the "controlling non-React widgets" escape hatch react.dev documents ([Synchronizing with Effects](https://react.dev/learn/synchronizing-with-effects)). PixiJS itself publishes no React guide for this, but it needs nothing Pixi-specific.
2. *@pixi/react 8.0.5:* an official custom react-reconciler "designed exclusively for React 19" (peer `react >= 19`), modeled on react-three-fiber, with an `extend()` catalogue API ([release post](https://pixijs.com/blog/pixi-react-v8-live); [npm](https://registry.npmjs.org/@pixi/react/latest); [extend docs](https://react.pixijs.io/extend/)). It is maintained (releases through 2025-12, React 19.2 support merged 2026-05) but much less exercised than Pixi itself (75k vs 867k weekly downloads) and has a live GPU-memory-leak issue tied to its `draw` prop pattern (open since 2026-04, [#648](https://github.com/pixijs/pixi-react/issues)) ([repo](https://github.com/pixijs/pixi-react); [downloads](https://api.npmjs.org/downloads/point/last-week/@pixi/react)).

**Testing.** Not first-party solved. Headless/Node use requires a custom `DOMAdapter` ([environments guide](https://pixijs.com/8.x/guides/concepts/environments)) or the community `pixijs-userland/node` package ([repo](https://github.com/pixijs-userland/node)); GitHub discussions confirm automated testing is a recurring pain point ([#10788](https://github.com/pixijs/pixijs/discussions/10788)). The realistic CI story is a real browser: Playwright screenshots or Vitest browser mode (§3), with the experimental canvas renderer as a determinism escape hatch if headless WebGL proves flaky.

### 2.2 Phaser 4

**Health.** Phaser 4.0.0 went stable 2026-04-10 after a long RC series; 4.2.1 published 2026-07-09; ~monthly releases; 40k stars ([npm time field](https://registry.npmjs.org/phaser); [v4.0.0 release](https://phaser.io/download/release/v4.0.0); [repo](https://github.com/phaserjs/phaser)). 274k weekly downloads ([downloads API](https://api.npmjs.org/downloads/point/last-week/phaser)).

**Pixel art is genuinely first-class.** `render: { pixelArt: true }` sets nearest-neighbor filtering, disables antialiasing, and sets `roundPixels: true` (now the default) in one flag ([core Config API](https://docs.phaser.io/api-documentation/class/core-config)); Phaser 4 adds `smoothPixelArt: true` for scaled/rotated pixel art, documented in a dedicated first-party pixel-art guide ([Phaser 4 Pixel Art Guide](https://github.com/phaserjs/phaser/blob/master/docs/Phaser%204%20Pixel%20Art%20Guide/Phaser%204%20Pixel%20Art%20Guide.md)).

**The cost is the monolith.** The package unconditionally bundles Arcade and Matter physics, input, audio, loader, tilemaps, particles — 355 KB gzip, not designed to be tree-shaken ([bundlephobia](https://bundlephobia.com/package/phaser); [physics docs](https://docs.phaser.io/phaser/concepts/physics)). For a server-authoritative renderer, most of that is dead weight. It is JS with authoritative hand-maintained `.d.ts`, ESM build included ([npm](https://registry.npmjs.org/phaser/latest)).

**React story.** No reconciler; the officially blessed pattern is the `template-react-ts` template (Phaser 4 + React 19 + Vite): a `PhaserGame.tsx` that imperatively mounts `Phaser.Game`, plus a plain `EventBus` emitter for React↔scene communication ([template-react-ts](https://github.com/phaserjs/template-react-ts); [template-react README](https://github.com/phaserjs/template-react)). Workable, but it's the same imperative mount we'd write for Pixi — with a full game framework (scene manager, its own loop, loader) between us and the sprites.

**Testing.** `renderType: Phaser.HEADLESS` exists but "still absolutely relies on the DOM being present and available. This mode is meant for unit testing, not for running Phaser on the server" ([Phaser.HEADLESS docs](https://newdocs.phaser.io/docs/3.55.2/focus/Phaser.HEADLESS)) — so logic tests need at least jsdom/browser, visuals need a real browser.

### 2.3 Excalibur

Pre-1.0: v0.32.0 (2025-12-23), active development (pushed 2026-07-19, commits 46 of the last 52 weeks) but no 1.0 shipped ([releases](https://github.com/excaliburjs/excalibur/releases); [repo](https://github.com/excaliburjs/Excalibur)). 15.5k weekly downloads — ~2% of Pixi's ([downloads API](https://api.npmjs.org/downloads/point/last-week/excalibur)). Strong pixel-art preset (`pixelArt: true` enables a sub-pixel shader; docs explain why bare `antialiasing: false` shimmers — [pixel-art docs](https://excaliburjs.com/docs/pixel-art/)), a pleasant fluent Actions/easing API ([actions docs](https://excaliburjs.com/docs/actions/)), and a first-party Aseprite plugin that reads native `.aseprite` files ([plugin docs](https://excaliburjs.com/docs/aseprite-plugin/)). But: no React integration or guidance (embedding is DIY via `canvasElement` — [EngineOptions API](https://excaliburjs.com/api/interface/EngineOptions/)), the `@excaliburjs/testing` package is stale (last push 2024-09 — [repo](https://github.com/excaliburjs/excalibur-testing)), and pre-1.0 + tiny community is real API-churn risk. TypeScript-native, ESM, 145 KB gzip, zero deps ([npm](https://registry.npmjs.org/excalibur); bundlephobia).

### 2.4 Kaplay

Disqualified for this app shape. Stable is v3001.0.19 from June 2025; v4000 missed its Q1-2026 roadmap target and is still alpha ([npm](https://registry.npmjs.org/kaplay); [roadmap wiki](https://github.com/kaplayjs/kaplay/wiki/KAPLAY-Roadmap-2026)). Decisively, a maintainer states verbatim: "IT JUST DOESN'T WORK. DON'T USE REACT AND KAPLAY AT THE SAME TIME ON THE SAME PAGE IF IT'S A SINGLE PAGE APP," recommending an iframe + `postMessage` boundary instead ([discussion #597](https://github.com/kaplayjs/kaplay/discussions/597)). An iframe wall between the match scene and the React HUD is the opposite of our interop requirement. (Otherwise fine on paper: `crisp: true`, `texFilter`, `tween()`/`animate()`, 67 KB gzip — [KAPLAYOpt](https://kaplayjs.com/docs/api/KAPLAYOpt/); [animation guide](https://kaplayjs.com/docs/guides/animation/).)

### 2.5 Hand-rolled Canvas 2D

Everything needed is standard platform API: `drawImage` 9-arg sprite slicing ([MDN drawImage](https://developer.mozilla.org/en-US/docs/Web/API/CanvasRenderingContext2D/drawImage)), `imageSmoothingEnabled = false`, a `requestAnimationFrame` loop (use the callback timestamp so speed is refresh-rate-independent — [MDN rAF](https://developer.mozilla.org/en-US/docs/Web/API/Window/requestAnimationFrame)) inside the react.dev-documented effect-with-cleanup pattern ([Synchronizing with Effects](https://react.dev/learn/synchronizing-with-effects)), plus the §1 DPR dance. `OffscreenCanvas`/worker rendering is Baseline since March 2023 if ever needed ([MDN OffscreenCanvas](https://developer.mozilla.org/en-US/docs/Web/API/OffscreenCanvas)). Zero dependencies, trivially deterministic, and node-canvas can even back jsdom's 2D context for unit tests ([jsdom README](https://github.com/jsdom/jsdom#canvas-support)). The catch: the prototype needs ~100 lines, but the full match view means hand-building a scene graph, z-ordering, animation bookkeeping, culling, and text — i.e. re-deriving the 20% of Pixi we'd actually use, without its ecosystem.

### 2.6 DOM/CSS — React is the renderer

For auto-battler entity counts this is more credible than game-dev folklore suggests:

- Idle/attack sprite-sheet animation is pure CSS: `animation-timing-function: steps(n)` stepping `background-position` ([MDN steps()](https://developer.mozilla.org/en-US/docs/Web/CSS/easing-function/steps)); crispness via `image-rendering: pixelated` (§1).
- Unit movement from server ticks maps to `transform: translate(...)` transitions — `transform` and `opacity` are the two properties browsers composite off the main thread ([web.dev compositor-only properties](https://web.dev/articles/stick-to-compositor-only-properties-and-manage-layer-count); [web.dev animations guide](https://web.dev/articles/animations-guide)), and tick-rate position updates (a few Hz) are far below React re-render pain thresholds; React batches state updates per handler ([react.dev queueing updates](https://react.dev/learn/queueing-a-series-of-state-updates)).
- HUD interop is free — health bars, tooltips, and damage numbers are siblings in the same tree, no coordinate bridging.
- Testing is the best of any option: plain DOM assertions in Vitest, screenshots without any GPU question.

Hard ceilings, kept factual: no shaders/palette-swap/lighting effects (CSS has no shader primitive; filters are coarse), each independently animated node wants its own compositor layer and web.dev explicitly warns to "manage layer count" (same article), and per-frame imperative control (hit-stop, screen-shake curves, particles) fights the declarative model. One documented unknown: no MDN/spec text explicitly confirms `image-rendering: pixelated` semantics under `transform: scale()` (it demonstrably works in practice, but the spec is silent — [MDN image-rendering](https://developer.mozilla.org/en-US/docs/Web/CSS/image-rendering)); prefer integer-sized layout scaling over transform-scaling for sprites.

## 3. Testability in CI (all canvas options)

- **Playwright visual comparisons** (`toHaveScreenshot`) are the workhorse; docs warn baselines must be generated in the same environment they're compared in ("Browser rendering can vary based on the host OS… headless mode"), and support `maxDiffPixels` tolerance and `animations: "disabled"` to freeze CSS animations for determinism ([Playwright test-snapshots](https://playwright.dev/docs/test-snapshots); [PageAssertions](https://playwright.dev/docs/api/class-pageassertions)).
- **Vitest browser mode is stable as of Vitest 4** (experimental tag removed), Playwright-provider recommended, and ships a `toMatchScreenshot` matcher — so component-level canvas tests can run in a real headless browser inside the normal test runner ([Vitest 4 announcement](https://vitest.dev/blog/vitest-4); [browser mode guide](https://vitest.dev/guide/browser/)).
- **Headless WebGL** runs on a software rasterizer; Playwright has no official doc on GPU behavior — the evidence lives in issue trackers ([playwright#15533](https://github.com/microsoft/playwright/issues/15533); [Chromium issue 40540071](https://issues.chromium.org/issues/40540071)). Expect screenshot baselines to be environment-pinned (Docker) for any WebGL option; Pixi's canvas renderer preference is a fallback if that bites.
- **jsdom has no canvas** unless the `canvas` (node-canvas) package is installed as a peer, and even then 2D only, no WebGL ([jsdom README](https://github.com/jsdom/jsdom#canvas-support)). Corollary for architecture: keep match-view *state* (tick interpolation, animation state machines) in plain TS modules that test in Node with no renderer at all, and treat the render layer as a thin projection.

## 5. Recommendation (ranked)

1. **PixiJS v8, mounted imperatively (`useRef` + `useEffect`), without @pixi/react — build the first prototype on this.** It is the only candidate that is exactly the abstraction we need (scene graph + sprites + sprite-sheet animation; no bundled physics/input), the healthiest by every maintenance signal, TypeScript/ESM-native so it fits the raw-TS + Vite conventions with zero build weirdness, and its pixel-art/DPR knobs are first-class. The imperative mount keeps React's job (HUD, tooltips, routing) and the render loop's job (ticker-driven interpolation of server ticks) cleanly separated, using only the escape-hatch pattern react.dev itself documents. Skip @pixi/react for now: it's React-19-compatible but adds a reconciler layer with 1/10th the usage of Pixi itself and an open GPU-memory-leak issue, and per-frame network-driven updates don't want to flow through React anyway.
2. **DOM/CSS (React as the renderer) — credible runner-up, and the fastest possible prototype.** For two idle characters it is unbeatable: `steps()` sprite animation + `image-rendering: pixelated`, perfect HUD interop, best-in-class testability, zero dependencies. It remains viable surprisingly deep into auto-battler scope (compositor-driven `transform` movement at tick rate). It loses only on the effects ceiling — no shaders/particles/palette swaps — which an auto-battler will eventually want.
3. **Hand-rolled Canvas 2D** — right primitives, but for the full match view it converges on rebuilding Pixi's core with no ecosystem; only worth it if we want absolute determinism and minimal deps more than velocity.
4. **Phaser 4** — excellent pixel-art story and healthy post-4.0 cadence, but a 355 KB monolith whose scene manager, loader, physics, and input solve problems a server-authoritative renderer doesn't have; its own React pattern is the same imperative mount we'd write for Pixi.
5. **Excalibur** — nicest authoring API and Aseprite-native pipeline, but pre-1.0, ~2% of Pixi's adoption, stale testing tooling, no React story.
6. **Kaplay** — ruled out by the maintainer's own guidance against React SPA embedding (iframe isolation would sever the HUD interop requirement) plus a year-stale stable release.

**Prototype shape (two characters, idle animation):** a `MatchView` React component owning a Pixi `Application` (`init({ resolution: devicePixelRatio, autoDensity: true, roundPixels: true })`, `TextureStyle.defaultOptions.scaleMode = 'nearest'`), loading one sprite sheet, playing two `AnimatedSprite`s; a sibling absolutely-positioned React HUD layer above the canvas; interpolation/animation state in a plain TS module unit-tested in Node; one Playwright/Vitest-browser screenshot test with animations frozen. This exercises every seam the real match view needs (mount lifecycle, DPR, asset pipeline, HUD layering, CI screenshots) for roughly the same effort as the DOM version.

## 6. Revisit triggers

Re-open this decision if any of the following happens:

- **The match view turns out to be mostly UI.** If prototype work shows units are really just health bars + portraits + tooltips with little scene-y motion, the DOM/CSS route (rank 2) wins on interop and testability — drop the canvas.
- **Declarative scene composition pressure.** If the imperative Pixi layer accretes React-shaped state plumbing (lots of props → scene sync code), re-evaluate @pixi/react — specifically after its GPU-context leak ([pixi-react #648](https://github.com/pixijs/pixi-react/issues)) is fixed and if its release cadence holds.
- **CI screenshot flakiness from headless WebGL.** If baselines won't stabilize even environment-pinned, switch the test renderer to Pixi's canvas preference ([8.16.0](https://pixijs.com/blog/8.16.0)) or reconsider DOM; if the canvas renderer's "experimental" label never graduates, weigh that too.
- **Effects ambitions exceed Canvas-2D-class rendering** (shaders, lighting, palette swaps, thousands of particles): confirms Pixi and rules out ever falling back to DOM — also the trigger to *stop* considering rank 2 a fallback.
- **Entity counts or update rates jump an order of magnitude** (e.g. replay scrubbing rendering many matches, or per-frame projectile swarms): DOM and raw Canvas 2D fall away; Pixi (WebGL/WebGPU) absorbs this without an architecture change.
- **Maintenance signals flip:** Pixi release cadence stalling or @pixi/react abandonment (watch [pixijs/pixijs](https://github.com/pixijs/pixijs) releases); conversely, Excalibur shipping 1.0 with a React story would make it worth a second look for its Aseprite pipeline.
- **We need server-side/Node rendering of matches** (thumbnails, replays rendered off-client): the DOMAdapter/community-node story ([environments guide](https://pixijs.com/8.x/guides/concepts/environments); [pixijs-userland/node](https://github.com/pixijs-userland/node)) must be validated before committing — today it is not first-party.
