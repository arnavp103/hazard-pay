# Agent framework survey: ADK, Mastra, Vercel AI SDK — borrow vs. build for `packages/agent`

- **Date:** 2026-07-19
- **Question:** How do Google's Agent Development Kit (ADK), Mastra, and the Vercel AI SDK structure their agent runtimes (agentic loop, tool invocation, state/persistence, streaming, multi-agent composition), and what should a future `packages/agent` borrow versus build for an event-sourced agent runtime on Postgres?
- **Context:** Hazard Pay's in-game "AI leaders" are real LLM agents with distinct prompts/tools/models. Stack: TypeScript monorepo, Postgres + Drizzle, Neverthrow, functional patterns. No build step; packages export raw TS.
- **Method:** Primary sources only — official docs and GitHub source of each framework, plus the two assigned readings. Every claim carries its source URL. Versions surveyed: ADK docs at adk.dev with `google/adk-python` on `main` (official TypeScript ADK exists, `@google/adk`); Mastra ≥ 1.0 (`@mastra/core` 1.51.x); Vercel AI SDK 7 (`ai@7.0.31`).

---

## 1. Framework architectures

### 1.1 Google ADK

**Loop structure.** ADK's runtime is a cooperative event loop. The `Runner` calls `agent.run_async(ctx)`; agent logic `yield`s `Event` objects one at a time; the Runner processes each event — committing its side effects via `SessionService`/`ArtifactService` — *before* signaling the agent generator to resume, then forwards the event upstream ([Runtime Event Loop](https://adk.dev/runtime/event-loop/index.md); [`runners.py`](https://github.com/google/adk-python/blob/main/src/google/adk/runners.py)). Inside an `LlmAgent`, the LLM-call ↔ tool-call cycle is driven by a `SingleFlow` request-processor pipeline; `AutoFlow` extends it with agent-transfer capability ([`single_flow.py`](https://github.com/google/adk-python/blob/main/src/google/adk/flows/llm_flows/single_flow.py); [`auto_flow.py`](https://github.com/google/adk-python/blob/main/src/google/adk/flows/llm_flows/auto_flow.py)).

**The Event object is the load-bearing abstraction.** `Event` extends `LlmResponse` and carries `author`, `invocation_id`, `id`, `timestamp`, `branch`, content, and an `EventActions` payload with `state_delta`, `artifact_delta`, `transfer_to_agent`, `escalate`, `requested_tool_confirmations`, `agent_state` (checkpoint data), and `rewind_before_invocation_id` ([`event.py`](https://github.com/google/adk-python/blob/main/src/google/adk/events/event.py); [`event_actions.py`](https://github.com/google/adk-python/blob/main/src/google/adk/events/event_actions.py)). The same object is simultaneously the streaming unit (forwarded to the UI) and the sole unit of persisted mutation: state changes ride on events as deltas, and the `SessionService` applies them when the event is appended ([Events](https://adk.dev/events/index.md)). Partial (streaming) events are forwarded upstream but *never* committed — `BaseSessionService.append_event` returns early `if event.partial` ([`base_session_service.py`](https://github.com/google/adk-python/blob/main/src/google/adk/sessions/base_session_service.py); [Runtime Event Loop](https://adk.dev/runtime/event-loop/index.md)).

**Tool model.** Function tools get an auto-injected `ToolContext` (session, state, actions) ([Function Tools](https://adk.dev/tools-custom/function-tools/index.md)). `LongRunningFunctionTool` returns an initial result and then *ends/pauses the agent run*; resumption correlates a later `FunctionResponse` with the original `function_call.id` via `event.long_running_tool_ids` (same page). Human-in-the-loop tool confirmation exists in two flavors — `FunctionTool(..., require_confirmation=True)` and `tool_context.request_confirmation(...)` — both experimental ([Action Confirmations](https://adk.dev/tools-custom/confirmation/index.md)).

**State & persistence.** A `Session` = event list + state map. `SessionService` implementations: `InMemorySessionService`, `DatabaseSessionService` (SQLAlchemy async engine; docs name `asyncpg` for PostgreSQL, with a two-tiered locking scheme serializing `append_event` per session), and `VertexAiSessionService` ([Session](https://adk.dev/sessions/session/index.md); [`database_session_service.py`](https://github.com/google/adk-python/blob/main/src/google/adk/sessions/database_session_service.py)). State keys are scoped by prefix — `app:`, `user:`, and `temp:` (never persisted; trimmed from the event's `state_delta` before storage) ([`state.py`](https://github.com/google/adk-python/blob/main/src/google/adk/sessions/state.py); [State](https://adk.dev/sessions/state/index.md); [`base_session_service.py`](https://github.com/google/adk-python/blob/main/src/google/adk/sessions/base_session_service.py)). Invocation resumability is explicit: `App(resumability_config=ResumabilityConfig(is_resumable=True))` allows pausing on a long-running function call and resuming "from the last event, if it's paused or failed midway through," with an at-least-once tool re-execution guarantee and loss of `temp:` state on resume ([`_configs.py`](https://github.com/google/adk-python/blob/main/src/google/adk/apps/_configs.py); [Resume Agents](https://adk.dev/runtime/resume/index.md)). Workflow agents resume from recorded position (`current_sub_agent`, `times_looped`, completed parallel branches) (same page). Long-term memory (`MemoryService`) and binary artifacts (`ArtifactService`) are separate pluggable services ([`base_memory_service.py`](https://github.com/google/adk-python/blob/main/src/google/adk/memory/base_memory_service.py); [Context](https://adk.dev/context/index.md)).

**Streaming.** `RunConfig.streaming_mode` ∈ {NONE, SSE, BIDI}; SSE yields partial events through `run_async`, while bidirectional (live audio/video) uses `runner.run_live()`, which yields some events it deliberately never persists (inline audio blobs) ([Runtime Config](https://adk.dev/runtime/runconfig/index.md); [`runners.py`](https://github.com/google/adk-python/blob/main/src/google/adk/runners.py)).

**Multi-agent.** Single-parent `sub_agents` tree; LLM-driven delegation via a synthesized `transfer_to_agent(agent_name=...)` function call intercepted by `AutoFlow`; deterministic orchestration via `SequentialAgent`/`ParallelAgent`/`LoopAgent` (loop exits on `max_iterations` or an event with `actions.escalate=True`); `AgentTool` wraps an agent as a callable tool that forwards state/artifact deltas back to the parent context ([Custom Agents](https://adk.dev/agents/custom-agents/index.md); [Template Agent Workflows](https://adk.dev/agents/workflow-agents/index.md)). Docs note that in ADK 2.0 (Python/Go) template workflow agents are being superseded by graph-based workflows (same page). Cross-cutting hooks: before/after agent/model/tool callbacks per agent, and Runner-global plugins ([Callbacks](https://adk.dev/callbacks/index.md); [Plugins](https://adk.dev/plugins/index.md)).

**Language support.** Officially Python, TypeScript, Go, Java, Kotlin; the TypeScript SDK is `@google/adk` at v0.2.0 — far younger than the Python line ([Event Loop version badges](https://adk.dev/runtime/event-loop/index.md); [TypeScript Quickstart](https://adk.dev/get-started/typescript/index.md)).

### 1.2 Mastra

**Loop structure.** Mastra (TypeScript; 1.0 released 2026-01-20 — [changelog](https://mastra.ai/blog/changelog-2026-01-20)) does *not* depend on the public `ai` package at runtime; it vendors AI SDK v4/v5/v6/v7 source into private internal packages ("Package to vendor ai-sdk v5 into our codebase so we don't have to use it in any package as a dependency" — [`packages/_vendored/ai_v5/README.md`](https://github.com/mastra-ai/mastra/blob/main/packages/_vendored/ai_v5/README.md)) and implements its own loop: `packages/core/src/loop/loop.ts` builds a `LoopRun` and executes it via `workflowLoopStream` — **the agentic loop is itself a workflow run on Mastra's workflow engine** ([`loop.ts`](https://github.com/mastra-ai/mastra/blob/main/packages/core/src/loop/loop.ts)). Loop controls: `maxSteps` (default 5), `stopWhen`, `prepareStep`, per-iteration `onIterationComplete` (return `{continue, feedback}`), scorer-gated `isTaskComplete`, and `toolCallConcurrency` ([Agent.generate() reference](https://mastra.ai/reference/agents/generate); [Supervisor agents](https://mastra.ai/docs/agents/supervisor-agents)).

**Tool model.** `createTool({id, description, inputSchema, outputSchema, execute})` with any Standard JSON Schema library (Zod, Valibot, ArkType) ([createTool reference](https://mastra.ai/reference/tools/create-tool); [Using Tools](https://mastra.ai/docs/agents/using-tools)). MCP client and server via `@mastra/mcp`, including per-request dynamic toolsets for multi-tenant credentials ([MCP overview](https://mastra.ai/docs/mcp/overview)). Human-in-the-loop is deep: `requireApproval` per tool or `requireToolApproval` per call emits a `tool-call-approval` chunk (stream) or `finishReason: 'suspended'` (generate), resumed via `approveToolCall`/`declineToolCall`; a tool's `execute` can also call `suspend()` mid-flight with typed suspend/resume schemas. Approval suspension is backed by persisted snapshots — "Agent approval uses snapshots to capture request state… deleted once the run finishes" — and `listSuspendedRuns()` rediscovers pending runs after a process restart ([Agent approval](https://mastra.ai/docs/agents/agent-approval)). Approvals propagate up arbitrarily deep supervisor→subagent chains (same page).

**State & persistence.** Memory (`@mastra/memory`) requires a storage adapter; Postgres is first-class via `@mastra/pg`, alongside LibSQL, MongoDB, Upstash, and ~20 more backends under `/stores/` ([Memory overview](https://mastra.ai/docs/memory/overview); [GitHub `/stores/`](https://github.com/mastra-ai/mastra/tree/main/stores)). Conversation identity is `thread` (session) + `resource` (stable owner); resuming a conversation is just calling `.generate()`/`.stream()` again with the same identifiers ([Message history](https://mastra.ai/docs/memory/message-history); [Memory overview](https://mastra.ai/docs/memory/overview)). Memory features: recent-history injection (default `lastMessages: 10`), working memory (Markdown or schema-typed, thread- or resource-scoped), semantic recall (vector search over past messages), and observational memory (background compression) ([Working memory](https://mastra.ai/docs/memory/working-memory); [Semantic recall](https://mastra.ai/docs/memory/semantic-recall)).

**Durable execution is the standout.** Workflows (`createWorkflow`/`createStep` with `.then/.parallel/.branch/.dountil/.foreach`) suspend via `await suspend(payload)` inside a step; the run's state becomes a **snapshot** — runId, status, per-step context, execution path, suspended paths, workflow state — serialized to a `workflow_snapshot` table in the configured store; `run.resume({step, resumeData})` reloads the snapshot and continues, callable "from anywhere… including HTTP endpoints, event handlers… or timers" ([Suspend and resume](https://mastra.ai/docs/workflows/suspend-and-resume); [Snapshots](https://mastra.ai/docs/workflows/snapshots)). The Postgres adapter implements `persistWorkflowSnapshot`/`loadWorkflowSnapshot` as raw SQL against that table ([`stores/pg/.../workflows/index.ts`](https://github.com/mastra-ai/mastra/blob/main/stores/pg/src/storage/domains/workflows/index.ts)). Note the model: **persisted state is a mutable point-in-time snapshot, not an append-only event log** — resume restores the latest snapshot; there is no replay-from-history. Orphaned runs restart via `restartAllActiveWorkflowRuns()` ([Workflows overview](https://mastra.ai/docs/workflows/overview)). Workflows can alternatively run on Inngest ([Workflow Runners](https://mastra.ai/docs/deployment/workflow-runners)).

**Streaming.** `.stream()` emits Mastra-native `ChunkType` events (`text-delta`, `reasoning-*`, `tool-call`, `tool-call-delta`, `tool-result`, `tool-call-approval`, `step-start/finish`, workflow lifecycle events), each tagged with `runId` and `from: AGENT|USER|SYSTEM|WORKFLOW`; `@mastra/ai-sdk` converts to AI SDK v5-compatible streams ([ChunkType reference](https://mastra.ai/reference/streaming/ChunkType); [Streaming guide](https://mastra.ai/guides/concepts/streaming)).

**Multi-agent.** The current pattern is supervisor agents: an `agents: {...}` map auto-converts subagents into `agent-<key>` tools, with delegation hooks (`onDelegationStart/Complete`, `messageFilter`), memory isolation per subagent (fresh ephemeral thread + deterministic derived `resourceId`), and background subagent tasks ([Supervisor agents](https://mastra.ai/docs/agents/supervisor-agents)). The older `AgentNetwork` is deprecated in its favor ([Agent networks](https://mastra.ai/docs/agents/networks); [migration guide](https://mastra.ai/guides/migrations/network-to-supervisor)). Agents also run as workflow steps with usage aggregation ([Workflows overview](https://mastra.ai/docs/workflows/overview)).

### 1.3 Vercel AI SDK (v7)

**Loop structure.** The SDK's agent is a *typed tool loop*, nothing more. `ToolLoopAgent` (stable since v6; hardened in v7, `ai@7.0.31` as of 2026-07-17 — [AI SDK 7 announcement](https://vercel.com/blog/ai-sdk-7); npm release history via `npm view ai time`) wraps the `generateText`/`streamText` multi-step loop. `Agent` itself is just an interface (`version: 'agent-v1'`) you can implement yourself ([Agent interface reference](https://ai-sdk.dev/docs/reference/ai-sdk-core/agent); [`tool-loop-agent.ts`](https://github.com/vercel/ai/blob/main/packages/ai/src/agent/tool-loop-agent.ts)). The loop ends when (verbatim from source): "The model returns a finish reason other than `tool-calls`; A tool without an execute function is called; A tool call needs approval; One of the provided stop conditions returns `true`" ([`stop-condition.ts`](https://github.com/vercel/ai/blob/main/packages/ai/src/generate-text/stop-condition.ts)). Built-in conditions: `isStepCount` (default 20 for `ToolLoopAgent`), `hasToolCall`, `isLoopFinished`; `prepareStep` overrides model/tools/messages per step (supporting context compaction); the docs explicitly recommend hand-rolling a `while` loop over `responseMessages` when you want full control ([Loop Control](https://ai-sdk.dev/docs/agents/loop-control); [Manual Agent Loop cookbook](https://ai-sdk.dev/cookbook/node/manual-agent-loop)).

**Tool model.** `tool({description, inputSchema, execute?, contextSchema?, toModelOutput?})` with Zod or JSON Schema; `execute` is optional *by design* "because you might want to forward tool calls to the client or to a queue instead of executing them in the same process" ([Tool Calling](https://ai-sdk.dev/docs/ai-sdk-core/tools-and-tool-calling)). `dynamicTool()` covers schema-unknown-at-compile-time tools ([Dynamic Tools](https://ai-sdk.dev/docs/ai-sdk-core/tools-and-tool-calling#dynamic-tools)). `repairToolCall` (stable as of 7.0.20) fixes malformed tool calls ([Tool Call Repair](https://ai-sdk.dev/docs/ai-sdk-core/tools-and-tool-calling#tool-call-repair)). Tool approvals are a first-class subsystem: per-tool or generic `toolApproval` functions; a paused call surfaces `tool-approval-request` parts, the app records a decision as a `tool-approval-response` message, and a *second* invocation executes or denies — with optional HMAC signing (`experimental_toolApprovalSecret`) so a client can't forge approvals ([Tool Approvals](https://ai-sdk.dev/docs/agents/tool-approvals)). MCP support moved to `@ai-sdk/mcp` (`createMCPClient`), including tool-drift fingerprinting where the SDK pointedly does not persist baselines: "Core stays unopinionated… those are your app's responsibility" ([MCP docs](https://ai-sdk.dev/docs/ai-sdk-core/mcp-tools)).

**State & persistence.** Confirmed: the SDK stores nothing. It defines two message formats — `UIMessage` (persist this) vs `ModelMessage` (wire format), bridged by `convertToModelMessages` — and gives you an `onEnd` callback in `toUIMessageStream` where *you* call your own `saveChat()` ([Chatbot Message Persistence](https://ai-sdk.dev/docs/ai-sdk-ui/chatbot-message-persistence); [convertToModelMessages](https://ai-sdk.dev/docs/reference/ai-sdk-ui/convert-to-model-messages)). `result.consumeStream()` is required to guarantee persistence fires on client disconnect (same page). Stream resumption (`useChat({id, resume: true})`) requires the app to supply Redis-backed storage via the third-party `resumable-stream` package: "The AI SDK provides tools to connect to storage, but you need to set up the storage yourself" ([Chatbot Resume Streams](https://ai-sdk.dev/docs/ai-sdk-ui/chatbot-resume-streams)). Durability beyond streams lives in a separate `WorkflowAgent` (`@ai-sdk/workflow`) tied to Vercel's Workflow runtime ([WorkflowAgent](https://ai-sdk.dev/docs/agents/workflow-agent)).

**Streaming.** `streamText`'s `stream` (formerly `fullStream`, now deprecated) is an `AsyncIterable<TextStreamPart>` covering text/reasoning start-delta-end, tool-input start/delta/end, `tool-call`, `tool-result`, `tool-error`, `tool-approval-request/response`, `start-step`/`finish-step`, `abort`, `error`, `raw` ([streamText reference](https://ai-sdk.dev/docs/reference/ai-sdk-core/stream-text); [`stream-text-result.ts`](https://github.com/vercel/ai/blob/main/packages/ai/src/generate-text/stream-text-result.ts)). The UI Message Stream protocol is SSE with typed parts including app-defined `data-*` parts ([Stream Protocols](https://ai-sdk.dev/docs/ai-sdk-ui/stream-protocol)).

**Multi-agent.** Deliberately not a framework feature. Orchestration patterns (routing, parallel, orchestrator-worker, evaluator-optimizer) are documented as plain TypeScript adapted from Anthropic's "Building Effective Agents" ([Workflow Patterns](https://ai-sdk.dev/docs/agents/workflows)); composition is subagents-as-tools, written by hand ([Subagents](https://ai-sdk.dev/docs/agents/subagents)). No handoff primitive exists. Provider abstraction is the `LanguageModelV4` spec (V2/V3 still re-exported) with composable middleware via `wrapLanguageModel` ([`language-model/index.ts`](https://github.com/vercel/ai/blob/main/packages/provider/src/language-model/index.ts); [Middleware](https://ai-sdk.dev/docs/ai-sdk-core/middleware)); telemetry is OTel via `registerTelemetry` (`@ai-sdk/otel`) ([Telemetry](https://ai-sdk.dev/docs/ai-sdk-core/telemetry)).

## 2. The two assigned readings

### 2.1 "The Agentic Loop: Three Loops in a Trenchcoat" (bobbytables.io)

Source: https://www.bobbytables.io/p/the-agentic-loop-three-loops-in-a (resolved as given; author Robert Ross). The essay decomposes an agent runtime into three nested loops:

1. **Inference loop** — talks to the model. Its "three responsibilities: Make chat completion API calls…, Pass a tool usage request to your tool loop, [and] Manage chat history persistence." Persistence lives here because provider APIs are stateless: "You need to provide the entire conversation every time." Storage form is deliberately open: "This can be an array, a database table, Redis keys, whatever."
2. **Tool loop** — executes tool calls and feeds results back as ordinary content. Practical guidance: handle hallucinated/malformed tool calls defensively, correlate calls and results by tool-call ID, and return tool errors as content (not exceptions) so the model can self-correct.
3. **Human loop** — really "a blocking function call," not a loop: pause for approval/steering before continuing. Because that pause can last arbitrarily long, the essay recommends durable execution infrastructure (it cites Temporal) rather than an in-memory wait, and gating sensitive/irreversible operations behind it.

Streaming is bracketed out as orthogonal: "the shape of these stays the same" with or without token streaming.

All three frameworks are recognizable in this frame: the AI SDK ships loops 1–2 plus approval plumbing and leaves persistence out; Mastra ships all three with snapshot-backed durability for loop 3; ADK fuses loops 1–2 into its event loop and handles loop 3 via tool confirmation + invocation resumability.

### 2.2 "The Log is the Agent" (arXiv:2605.21997)

The assigned arXiv ID **resolves as given** — it is not a dud. Yohei Nakajima, *"The Log is the Agent: Event-Sourced Reactive Graphs for Auditable, Forkable Agentic Systems,"* submitted 2026-05-21, cs.AI/cs.MA ([abstract](https://arxiv.org/abs/2605.21997); [HTML full text](https://arxiv.org/html/2605.21997); code: [yoheinakajima/activegraph](https://github.com/yoheinakajima/activegraph)). Core claims, from the abstract and paper:

- **The append-only event log is the source of truth; the working graph is a deterministic projection** of the log, "recomputed by a replay operation that is guaranteed deterministic." Load, fork, and validate are all the same replay operation.
- **Reactive behaviors, not orchestration:** functions/LLM routines subscribe to graph patterns and emit new events; "No component instructs another; coordination happens entirely through the shared graph."
- **Determinism contract:** behaviors may not read wall-clock time, randomness, or fresh UUIDs directly, and model calls are recorded as events; on replay the recorded response is served from a content-addressed cache instead of re-invoking the model. Strict mode surfaces divergence; permissive mode allows fresh calls after edits.
- **Payoffs:** deterministic replay of any run, cheap forking at any event (shared prefix replays from cache, no re-billing — e.g., editing a prompt at step 150 of 200 re-executes only 150+), and end-to-end lineage from goal to the individual model call behind each artifact.

## 3. How an event-sourced approach maps onto each framework

**ADK: already event-sourced at the session level, but not replayable.** ADK satisfies the paper's first property structurally: the event is the atomic unit of both transport and persistence, state is only mutated via `state_delta`s carried on committed events, and `Session.state` is effectively a fold over the event list ([Events](https://adk.dev/events/index.md); [`base_session_service.py`](https://github.com/google/adk-python/blob/main/src/google/adk/sessions/base_session_service.py)). Resumability even resumes "from the last event" ([`_configs.py`](https://github.com/google/adk-python/blob/main/src/google/adk/apps/_configs.py)). What ADK lacks versus the paper: no determinism contract, no content-addressed model-response cache, so no byte-reproducible replay or cheap forking — the log records what happened but cannot deterministically re-derive it. `rewind_before_invocation_id` in `EventActions` hints at log-rewind mechanics ([`event_actions.py`](https://github.com/google/adk-python/blob/main/src/google/adk/events/event_actions.py)). ADK is the closest architectural template for an event-sourced runtime.

**Mastra: durable, but snapshot-sourced, not event-sourced.** Mastra persists a *mutable point-in-time snapshot* per run (`workflow_snapshot` table) and restores it on resume ([Snapshots](https://mastra.ai/docs/workflows/snapshots); [`stores/pg` source](https://github.com/mastra-ai/mastra/blob/main/stores/pg/src/storage/domains/workflows/index.ts)); approval snapshots are even "deleted once the run finishes" ([Agent approval](https://mastra.ai/docs/agents/agent-approval)). That gives excellent crash-recovery and human-in-the-loop pauses but none of the paper's replay/fork/lineage properties — there is no authoritative history to project from. Retrofitting event sourcing onto Mastra would mean fighting its storage layer. Its *stream*, however, is nearly an event log already: `ChunkType` events with `runId` and `from` attribution ([ChunkType](https://mastra.ai/reference/streaming/ChunkType)) are a good vocabulary to borrow for event-record design.

**Vercel AI SDK: unopinionated, therefore the best substrate.** The SDK persists nothing and streams everything: `TextStreamPart` already enumerates the step/tool/approval lifecycle as typed parts ([`stream-text-result.ts`](https://github.com/vercel/ai/blob/main/packages/ai/src/generate-text/stream-text-result.ts)), the loop's four exit conditions are explicit ([`stop-condition.ts`](https://github.com/vercel/ai/blob/main/packages/ai/src/generate-text/stop-condition.ts)), and the docs bless manual loop ownership ([Manual Agent Loop](https://ai-sdk.dev/cookbook/node/manual-agent-loop)). An event-sourced runtime can wrap each model step: append `step-requested`/`step-completed` events (with the full `ModelMessage` request as the content-address key, per the paper's cache design), fold the log into the message array before each call, and treat execute-less tools + the two-call approval flow as the natural persistence seam for human/game-driven pauses. Nothing in the SDK resists this; it was designed to leave these decisions to the app ("Core stays unopinionated" — [MCP docs](https://ai-sdk.dev/docs/ai-sdk-core/mcp-tools)).

---

## 4. Borrow vs. build for `packages/agent`

Hazard Pay's constraints sharpen the choice: TypeScript-native, Postgres + Drizzle already in the stack, Neverthrow/functional style, many concurrent long-lived agents (one per AI leader, distinct prompt/tools/model), game state as the shared world the agents act on, and a strong product case for replay/fork/audit (spectating a leader's reasoning, debugging a match, branching "what-if" simulations).

### Borrow (adopt as dependencies)

1. **Vercel AI SDK core as the model-call layer** — `generateText`/`streamText`, the `LanguageModelV4` provider spec (per-leader model diversity for free), `tool()` with Zod, `repairToolCall`, middleware, and OTel telemetry ([Middleware](https://ai-sdk.dev/docs/ai-sdk-core/middleware); [Telemetry](https://ai-sdk.dev/docs/ai-sdk-core/telemetry)). Use single-step calls (or `stopWhen: isStepCount(1)`-style stepping) so the loop boundary stays in our runtime, exactly as the SDK's own Manual Loop guidance describes ([Loop Control](https://ai-sdk.dev/docs/agents/loop-control)).
2. **The AI SDK's stream-part and message vocabulary** — `TextStreamPart`/UI Message Stream part types and the `UIMessage` vs `ModelMessage` split ([Stream Protocols](https://ai-sdk.dev/docs/ai-sdk-ui/stream-protocol)) — as the wire format toward game clients, so any AI-SDK-compatible frontend tooling works.
3. **Execute-less tools + the two-call approval shape** as the suspension mechanism: a game action is a tool without `execute`; the game engine resolves it and the runtime appends the result event, mirroring both the SDK's deferred-execution design ([Tool Calling](https://ai-sdk.dev/docs/ai-sdk-core/tools-and-tool-calling)) and ADK's `LongRunningFunctionTool` pause/correlate-by-call-id pattern ([Function Tools](https://adk.dev/tools-custom/function-tools/index.md)).

### Borrow (steal the design, not the code)

4. **ADK's event contract**: one `Event` type carrying author, invocation id, content, and an *actions* payload (`state_delta`, `escalate`, `transfer_to_agent`-style control signals); commit-before-continue ordering; partial events streamed but never persisted; `temp:`-prefixed state that never hits storage ([Runtime Event Loop](https://adk.dev/runtime/event-loop/index.md); [State](https://adk.dev/sessions/state/index.md)). This is the skeleton of the runtime.
5. **The paper's determinism contract and replay design**: model calls recorded as request/response event pairs; content-addressed response cache keyed on the request; strict vs. permissive replay; fork = replay shared prefix from cache ([arXiv:2605.21997](https://arxiv.org/abs/2605.21997)). This is what turns the log from an audit trail into a fork/replay engine — none of the three frameworks provides it.
6. **Mastra's suspended-run discoverability** (`listSuspendedRuns` after restart) and its `runId`/`from`-attributed chunk taxonomy ([Agent approval](https://mastra.ai/docs/agents/agent-approval); [ChunkType](https://mastra.ai/reference/streaming/ChunkType)) — in an event-sourced store this falls out of a query over terminal-less logs, but the API shape is worth copying.
7. **The three-loops decomposition** as module boundaries: inference loop (owns log-fold → model call → append), tool loop (owns execution, ID correlation, errors-as-content), human/game loop (a blocking seam implemented as suspension events, durable because the log *is* the wait state) ([bobbytables.io](https://www.bobbytables.io/p/the-agentic-loop-three-loops-in-a)).

### Build

8. **The event store on Postgres via Drizzle.** Append-only `agent_events` table (id, run/invocation id, seq, author, type, payload JSONB, occurred_at) with a uniqueness constraint on (run, seq) for optimistic append; projections (current messages, leader state) derived by folding, optionally checkpointed. No framework offers a Drizzle-native, append-only store: ADK's is SQLAlchemy/Python-shaped, Mastra's is its own managed mutable tables, the AI SDK has none.
9. **The loop itself** as a small pure-ish reducer: `(log) -> nextAction` where `nextAction ∈ {callModel, executeTool, awaitGame, transfer, done}`, with Neverthrow `Result`s for tool failures rendered back as content per the essay's errors-as-content rule. The AI SDK loop is ~replaceable by design; owning it is cheap and buys replay/fork.
10. **Multi-agent composition for the game's needs**: leaders are peers acting on shared game state, not a supervisor tree — closer to the paper's "coordination through the shared graph" than to Mastra's supervisor or ADK's `transfer_to_agent`. Build a thin scheduler that wakes each leader's runtime on relevant game events; borrow `AgentTool`/subagent-as-tool only if a leader ever needs private helpers.

### Rejected alternatives

- **Adopt Mastra wholesale:** rejected. Its snapshot persistence is the wrong substrate for replay/fork (see §3), it vendors its own AI SDK forks ([vendored packages](https://github.com/mastra-ai/mastra/tree/main/packages/_vendored/ai_v5)), and it owns its Postgres schema — a poor fit next to Drizzle-managed game tables. Its HITL/suspend UX remains the best reference design.
- **Adopt ADK TypeScript:** rejected for now. The architecture is the right one, but `@google/adk` is v0.2.x ([TS Quickstart](https://adk.dev/get-started/typescript/index.md)) against a Python-first project, and its persistence is service-shaped rather than schema-shaped — we would still build the Drizzle store, at which point the event contract is the valuable part, and that we can copy.

### Top recommendation

Build a small event-sourced runtime in `packages/agent`: ADK's event-loop semantics (event = unit of streaming *and* persistence, commit-before-continue) on an append-only Drizzle/Postgres log, with the ActiveGraph paper's determinism contract and content-addressed model-call cache for replay and forking; use the Vercel AI SDK strictly as the model/tool/stream layer beneath a loop we own. Mastra and ADK-TS are references, not dependencies.
